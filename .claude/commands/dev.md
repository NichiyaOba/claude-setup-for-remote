# /dev — マルチエージェント開発ワークフロー

ユーザーから与えられたタスクに対し、以下のマルチエージェントワークフローを実行してください。
あなたはオーケストレーター（指揮者）です。各フェーズで専門エージェントを起動し、フィードバックループを管理します。

## ワークフロー概要

```
[タスク入力]
    ↓
Phase 0: 調査フェーズ       — investigator がコードベースを調査
    ↓
Phase 1: 設計・計画フェーズ  — designer が設計書＋コミット計画を作成 → design-reviewer がレビュー
    ↓
Phase 2: 実装フェーズ       — implementer がコミット計画に従って実装 → implementation-reviewer がレビュー
    ↓
[完了]
```

---

## Phase 0: 調査フェーズ

### Step 0.1: 調査エージェントの起動

`investigator` エージェント（subagent_type: "investigator"）を起動してください。

**プロンプトに含める情報:**
- ユーザーが指定したタスク要件: `$ARGUMENTS`
- リポジトリのルートパス

**実行例:**
```
Agent(subagent_type="investigator", prompt="
以下のタスクを実装するために、リポジトリを調査してください。

## タスク要件
{ユーザーの入力}

## リポジトリ
{リポジトリパス}
")
```

### Step 0.2: 調査レポートの受け取り

investigator の出力（調査レポート）を受け取り、Phase 1 へ渡す。
調査レポートには以下が含まれる:
- リポジトリ構成
- 類似実装とそのパターン
- git履歴から把握した設計方針
- 変更が必要なファイルの影響範囲
- 注意事項・制約

---

## Phase 1: 設計・計画フェーズ

### Step 1.1: 設計エージェントの起動

`designer` エージェント（subagent_type: "designer"）を起動してください。

**プロンプトに含める情報:**
- ユーザーが指定したタスク要件: `$ARGUMENTS`
- Phase 0 で取得した調査レポートの全文
- リポジトリのルートパス
- 前回のレビューフィードバック（2回目以降のループの場合）

**実行例:**
```
Agent(subagent_type="designer", prompt="
以下のタスクの設計書とコミット計画を作成してください。

## タスク要件
{ユーザーの入力}

## 調査レポート
{Phase 0 の調査レポート全文}

## リポジトリ
{リポジトリパス}

## レビューフィードバック（修正依頼の場合）
{前回のレビュー結果}
")
```

**designer の出力に含まれるもの:**
- 設計書（概要・スコープ・アーキテクチャ・詳細設計・技術的決定事項）
- コミット計画（コミット順・対象ファイル・変更詳細・コミットメッセージ）

### Step 1.2: 設計レビューエージェントの起動

設計エージェントの出力を受け取ったら、`design-reviewer` エージェント（subagent_type: "design-reviewer"）を起動してください。

**プロンプトに含める情報:**
- designer が出力した設計書＋コミット計画の全文
- 元のタスク要件
- 調査レポート

### Step 1.3: レビュー判定の処理

レビュー結果の「判定」を確認してください：

- **APPROVED の場合**: ユーザーに設計書・コミット計画・レビュー結果のサマリーを報告し、Phase 2 へ進む
- **NEEDS_REVISION の場合**: レビューフィードバックを設計エージェントに渡して Step 1.1 に戻る

**ループ上限**: 最大 **3回** までリトライする。3回目のレビューでも NEEDS_REVISION の場合は：
1. これまでの設計書とレビュー結果をユーザーに提示する
2. `AskUserQuestion` ツールでユーザーに判断を仰ぐ（「このまま実装に進む」/「設計を手動で修正する」/「タスクを中止する」）

---

## Phase 2: 実装フェーズ

### Step 2.1: 実装エージェントの起動

`implementer` エージェント（subagent_type: "implementer"）を起動してください。

**プロンプトに含める情報:**
- 承認済みの設計書の全文
- 承認済みのコミット計画（コミット順・対象ファイル・変更詳細）
- 元のタスク要件
- リポジトリのルートパス
- 前回のレビューフィードバック（2回目以降のループの場合）

**実装エージェントへの指示:**
コミット計画に記載された順番・単位でコードを実装すること。
計画に記載されていない変更を勝手に追加しないこと。

### Step 2.2: 実装レビューエージェントの起動

実装エージェントの報告を受け取ったら、`implementation-reviewer` エージェント（subagent_type: "implementation-reviewer"）を起動してください。

**プロンプトに含める情報:**
- 実装エージェントの実装報告
- 承認済みの設計書
- 承認済みのコミット計画
- 元のタスク要件

### Step 2.3: レビュー判定の処理

レビュー結果の「判定」を確認してください：

- **APPROVED の場合**: ユーザーに実装結果とレビュー結果のサマリーを報告し、ワークフロー完了
- **NEEDS_REVISION の場合**: レビューフィードバックを実装エージェントに渡して Step 2.1 に戻る

**ループ上限**: 最大 **3回** までリトライする。3回目のレビューでも NEEDS_REVISION の場合は：
1. これまでの実装内容とレビュー結果をユーザーに提示する
2. `AskUserQuestion` ツールでユーザーに判断を仰ぐ（「このまま完了とする」/「手動で修正する」/「タスクを中止する」）

## 完了報告

ワークフロー全体が完了したら、以下のフォーマットで最終報告を出力してください：

```markdown
# ワークフロー完了報告

## タスク
[元のタスク要件]

## 調査フェーズ
- 発見した類似実装: [一覧]
- 影響ファイル数: [N件]

## 設計・計画フェーズ
- ループ回数: [N回]
- 最終判定: [APPROVED]
- 設計サマリー: [設計の概要]
- コミット数: [N件]

## 実装フェーズ
- ループ回数: [N回]
- 最終判定: [APPROVED]
- 変更ファイル: [ファイル一覧]

## 次のステップ
[推奨される後続作業があれば記載]
```

---

## 注意事項

- 各エージェントの起動時は `Agent` ツールを使用する
- エージェント間のコンテキスト（調査レポート・設計書・コミット計画・レビュー結果）は明示的にプロンプトに含めて引き渡す
- ユーザーへの中間報告は各フェーズの切り替え時に行う
- エラーが発生した場合はユーザーに報告し、対応を相談する
- `TodoWrite` ツールでワークフローの進捗を可視化する
